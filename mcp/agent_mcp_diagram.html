Agent vs. MCP — What Does Each Do?

Agent (the LLM)

  • Thinks — understands your intent
  • Plans — breaks the task into steps
  • Decides — which tools to call and when
  • Interprets — reads tool results
  • Composes — writes the final answer or code
  • Iterates — retries or changes approach on error

MCP Server (the Hands)

  • Exposes tools — typed, documented actions
  • Executes — runs the actual code / API call
  • Returns structured data — JSON the agent can use
  • Maintains state — persistent REPL, map layers
  • Standardized protocol — any agent can connect
  • Swappable — add/remove servers without changing the agent
Key insight: An Agent is an LLM that can use tools. MCP (Model Context Protocol) is the standard protocol for how those tools are described and called — like USB for AI. The agent does not need to know how Earth Engine works; it just calls run_code or inspect_asset and gets results back.

Step-by-Step: A Real geeViz Query

User asks: “How has land use changed near Salt Lake City from 1985 to 2024? Show me a sankey chart and a map.”
1
User sends message
Through any interface — Claude.ai chat, Claude Code terminal, Cursor, ADK web UI, etc. The message is plain English.
2
Agent geocodes the location
The LLM calls search_places("Salt Lake City, UT") to get coordinates, then uses run_code with sal.getUSCounties(point, state_abbr='UT') to get the county boundary. No need to search for boundary datasets — sal provides them directly.
3
Agent finds the right dataset
Calls search_datasets("LCMS land use") to find the LCMS collection, then inspect_asset("USFS/GTAC/LCMS/v2024-10") to get real band names, date range, and class properties. No guessing — the agent reads live metadata from Earth Engine.
4
Agent looks up the charting function
Calls search_functions(function_name="summarize_and_chart") to get the exact signature, parameter types, and return format. Now it knows chart_type='sankey', transition_periods=[1985, 2000, 2024] (flat list), and that the result is a dict with result['chart'] and result['df'].
5
Agent generates the sankey chart
Calls run_code with a script that loads LCMS, runs cl.summarize_and_chart(lcms, area, band_names='Land_Use', chart_type='sankey', transition_periods=[1985, 2000, 2024]), and saves the result with cl.save_chart_png(result['chart'], 'sankey.png'). The MCP server executes this in a persistent REPL, talks to Earth Engine, and returns the saved file path.
6
Agent adds a companion time lapse to the map
Another run_code call adds the LCMS data as an interactive time lapse with Map.addTimeLapse(lcms_subset, {'autoViz': True, 'canAreaChart': True, 'areaChartParams': {'sankey': True, ...}}). The viewer will let users draw polygons and generate their own sankey diagrams on the fly.
7
Agent validates layers before showing the map
Calls map_control(action="test_layers") — this runs getMapId() on every layer in parallel (~1–2 seconds) to catch bad bands, viz errors, or computation failures. All layers pass, so the agent proceeds.
8
Agent opens the map
Calls map_control(action="view") to open the interactive geeView map in the browser. The user sees the time lapse with a slider, area charting tools, and the sankey option built in.
9
Agent composes final response
The LLM summarizes the land use transitions, describes key trends (urban expansion, forest change), and presents the sankey chart image inline. The user sees a complete analysis — 8+ MCP tool calls, zero lines of code written by hand.

Where Can You Use Agents + MCP?

Chat UI
Connect MCP servers in project settings. Best for exploration and one-off questions.
Terminal Agent
Configure MCP in .mcp.json. Full file edits, git, and GEE in one flow.
IDE Chat
MCP tools alongside code editing tools. Best for writing and debugging GEE scripts.
Your Own Code
Build with the Claude API + MCP SDK. Embed GEE analysis in dashboards, bots, pipelines.
The power of MCP: The geeViz MCP server is written once and works in all of these interfaces. No separate plugin per IDE or chat UI. Any MCP-compatible agent can connect — today and in the future.

Why Does This Matter?

Without MCP

  • Agent can only generate code — you copy-paste and run it
  • Agent guesses function signatures from training data (often wrong)
  • No way to test code before giving it to you
  • Each IDE needs a custom plugin
  • Agent cannot see actual asset metadata or map results

With geeViz MCP

  • Agent runs code directly — tests it, iterates on errors
  • Agent looks up real signatures via search_functions
  • Code is validated before you see it
  • One server works everywhere (Claude, Cursor, VS Code...)
  • Agent inspects assets, views maps, exports data live
